Introduction

Plume is a design package designed for Chinese International School's Summer Programme. It evolves from a private Github package known as cis_summer and before that, cisss. The code package used to power the backend of the site is known as Phoenix and is currently under production. After debranding and removal of all client-associated items, the package will be opensourced on Github under a yet undecided name.

The default stylesheet, screen.css is used across all media. An additional stylesheet, print.css contains generic classes for printer-friendliness and is only used on printers.

A consistent design language

In an effort to reduce guesswork and confusion, the stylesheet, class and ID definitions are defined in past tense for any action verbs and singular form for any nouns.

Other than syntax, general clarity, usability and legibility are the three key characteristics to keep in mind when designing any page built with Plume. Here are a few things to also keep in mind for best practise uses:

  • The website shall be 100% W3C HTML5 and CSS3 compliant. Pre-CSS3 browser-specific selectors may be used where appropriate (eg: -moz-border-radius).
  • The website should be semantic. This means using a header tag for titles, cite tags for captions and citations, and list tags properly.
  • Ems shall be used over pixels where possible to avoid unpleasant surprises. This is part of a decision to move Plume as quickly as possible to a mobile-first and responsive design.
  • The page should not use more than 3 fonts. It starts to get confusing after that.
  • There should always be some sort of breadcrumb and navigation bar to provide spatial awareness with regards to the rest of the site.
  • Linguistically, the CIS Summer Programme will use 100% British spelling. This means we use "programme", "colour" and "enroll". Content-providers should be consistent with the site linguistic preferences. This also means that the site must use the <html lang="en-GB"> tag. Designers should note any linguistic preferences in the lang parameter.

Typography

One of the most overlooked design aesthetics in web sites today is typography: that is, the use of different fonts for different purposes within the site. Fonts can help the user subconsciously identify differences between title and text, important and trial and so much more.

Web-enabled fonts (via @font-face in CSS) make the website feel unique, well-curated and well-maintained. However, not everyone can use this, which means we still need fallback fonts.

Plume uses 2 main fonts, one for headings and another for general text.

Headings

For headings, we use Nobile from Google Web Fonts. It looks like this:

AaBbCcDdEeFfGgHhIiJjKkLlMmNnOoPpQqRrSsTtUuVvWwXxYyZz0123456789

This means that all heading tags, <h1> thru <h3> are by default styled with Nobile. The remaining header tags <h4> to <h6> are left to the designer's discretion.

Note In the event that Nobile can't be loaded, the fallback font is Helvetica Neue, then Helvetica.

You can force use of this font on any element using class="font-nobile", though it is not recommended.

Text

Plain paragraph text is styled using the usual <p> syntax. Links are coloured and will underline upon rollover.

There's a number of cool things you can do with text. You can make it less significant with class="muted", or smaller with class="small".
Of course you can go the other way and write things in bold font-face with <strong> and feature text using class="featured".

Alerts

An alert gives vital information about an action and draws the user's attention to it. Alerts should be particularly succinct and used appropriately to avoid notification fatigue.

This gives you some information. Use <div class="alert"> to create one.
A red alert class="alert alert-red" indicates that an action was performed but the response was not as expected.
Of course, a green alert class="alert alert-green" describes the details of a successful action.
Finally, a yellow alert class="alert alert-yellow" hints at potential problems ahead, or gives a warning to the user.

Badges

Unlike alerts, which are used to indicate results or possible outcomes of an action, badges gives general advice and information througout the page, regardless of whether they are related to actions. Badges require the use of the <span class="badge"> tag.

There are a number of badges which correspond to the same colouring scheme.

Badges do not encapsulate the descriptive text - alerts do that. These means they serve two different purposes. It's much better to "overload" on badges than it is to cause notification fatigue using alerts.

Shields

Shields were born more of necessity with the CIS Summer Programme than anything else. They are an action item and therefore have hover states.

Unordered lists have a disc-shaped bullet point and are initialized with <ul>. Ordered lists however (<ol>), must have a default class (default: class="numeral") to define its list style type.

By convention, lists should not be encapsulated in paragraph tags.

Forms & Labels

Forms are interactive. They should invite the user to interact with each input element, while at the same time turning attention to its label. The label gives the user instructions and should be placed, ideally, as close to the input element itself as possible.
